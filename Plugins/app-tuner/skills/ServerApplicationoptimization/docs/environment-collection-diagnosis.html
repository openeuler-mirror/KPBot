环境采集 + 环境诊断流程设计项

本文罗列当前 skill 在环境备份阶段实际采集的指标、采集方式、产物文件，以及环境诊断阶段检查的项目、判定逻辑和输出字段。当前依据为 backup_environment.sh、diagnose_environment.py 和 references/environment-diagnosis.md。

流程位置 采集项 诊断项 诊断确认 输出结构 验证与缺口

流程位置与执行顺序

环境采集与诊断属于 Phase 1 的环境基座门控，必须在服务健康检查和基线测试之前完成。诊断结果需要用户确认后，才允许进入后续服务健康检查和基线性能确认。当前顺序设计如下：

1. 环境备份

执行 scripts/backup_environment.sh <output-dir>，按 BIOS、硬件、软件/OS/内核分类沉淀只读快照，并生成汇总报告。

2. 环境诊断

执行 scripts/diagnose_environment.py，基于备份目录、当前系统和可选清单输出诊断 JSON。

3. 诊断结果确认

向用户展示阻塞项、降级项、补采建议和风险说明；用户确认后写入 environment_diagnosis_confirmation_status=confirmed。

4. 后续门控

诊断结果已确认后，才进入服务健康检查、目标实例身份确认、基线建立和瓶颈识别。

skills/kpbot-app-tuner/scripts/backup_environment.sh --output-dir ./output/env --non-interactive
# 如需通过 Redfish 采集 BIOS 当前配置：
skills/kpbot-app-tuner/scripts/backup_environment.sh --output-dir ./output/env \
  --bmc-host <BMC_IP> --bmc-user <BMC_USER> --bmc-pass '<BMC_PASS>'
python3 skills/kpbot-app-tuner/scripts/diagnose_environment.py \
  --backup-dir ./output/env \
  --output ./output/env/environment-diagnosis.json
启动问询要求：开始采集前需要询问用户是否提供 BMC IP、用户名、密码。提供时通过 Redfish 只读接口采集 BIOS 当前配置；未提供时跳过 Redfish 并在 bios-redfish.txt 和汇总报告中记录 skipped。诊断完成后必须等待用户确认；未确认、拒绝或要求重建时，不进入服务健康、目标实例身份确认、基线性能确认和瓶颈识别。

当前环境采集项与采集方式

当前环境采集脚本采用顺序只读命令执行。每类信息都有独立采集文件，并额外生成 environment-backup-report.html 汇总报告。每个采集动作会写入 command-manifest.txt，记录 RUN、OK、FAIL 或 SKIP。命令不存在时不会中断流程，而是记录 missing-command。

采集维度 产物文件 当前采集指标 / 内容 采集命令或来源 用途
环境备份汇总报告 environment-backup-report.html 按 BIOS 配置、硬件配置信息、软件配置、OS/内核参数、运行环境边界汇总关键片段。报告片段会对序列号、UUID、MAC、IP、密码字段做基础脱敏。 脚本汇总各独立采集文件生成。 作为环境备份的一页式入口；完整证据仍以同目录原始文件为准。
CPU / Cache / SMT / NUMA / 频率 hardware-cpu.txt，兼容文件 cpu-info.txt、numa-topology.txt 总 CPU 数、在线 CPU、SMT 线程数、socket/core、cache 大小、NUMA 节点、CPU 频率、governor、cpufreq policy。 uname -m、lscpu、lscpu -e、/proc/cpuinfo、/sys/devices/system/cpu/cpu0/cache、/sys/devices/system/cpu/cpufreq、cpupower frequency-info 判断平台架构、核心数、线程数、CPU 型号，为 CPU/NUMA/编译优化提供基础信息。
内存条 / DDR / 规格 / 速率 / 编号 / 厂商 hardware-memory.txt，兼容文件 memory-info.txt DIMM 信息、内存控制器、内存容量、可用量、swap、/proc/meminfo 明细；非 root 时保留权限不足输出。 dmidecode -t memory、lshw -class memory、free -h、cat /proc/meminfo 判断内存容量、swap、缓存、HugePage 相关状态，并支撑目标规格约束核对。
磁盘 SSD/HDD 类型 / 大小 / 厂商 / 型号 hardware-disk.txt，兼容文件 disk-info.txt 块设备详细信息、SSD/HDD rotational 标记、传输类型、型号、序列号、厂商、空间、挂载点、I/O scheduler、队列参数、smartctl 扫描。 lsblk -O、lsblk -d -o NAME,TYPE,SIZE,MODEL,SERIAL,VENDOR,ROTA,TRAN,...、df -h、mount、/sys/block/*/queue、smartctl --scan 识别数据目录所在介质、空间风险、挂载参数，以及后续磁盘瓶颈预筛上下文。
网卡型号 / 速率 / 厂商 / 驱动 hardware-nic.txt，兼容文件 nic-info.txt PCIe 网络设备、链路接口、IP 地址、网卡速率/协商/驱动、offload、ring、coalesce。逐接口 ethtool 带短超时保护。 lspci -nn、ip -br link、ip -br address、ifconfig -a、ethtool、ethtool -i/-k/-g/-c 确认压测组网、服务端网卡、容器/主机网络边界和网络瓶颈预筛对象。
OS 配置 / sysctl / NUMA balancing / THP / XPS os-config.txt，兼容文件 os-kernel.txt、thp-status.txt、hugepages-status.txt 内核版本、发行版、sysctl -a、NUMA balancing、透明大页、HugePages、ulimit、RPS/XPS、IRQ affinity、irqbalance、tuned profile。 uname -a、/proc/version、/etc/os-release、sysctl -a、/proc/sys/kernel/numa_balancing、/sys/kernel/mm/transparent_hugepage/*、/sys/class/net/*/queues/*/*ps_cpus 判断内核版本、发行版、补丁清单匹配和 perf/linux-tools 兼容性。
BIOS / 固件 bios-info.txt、bios-redfish.txt、可选 bios-redfish-*.json OS 侧 BIOS/DMI/sysfs 信息；用户提供 BMC IP/用户/密码时，采集 Redfish 根、Systems、Managers、System、Bios、Bios/Settings JSON。 dmidecode -t bios/system/baseboard、/sys/class/dmi/id/*、Redfish /redfish/v1、/Systems、/Bios、/Bios/Settings 为 BIOS 高性能配置诊断提供证据。未提供 BMC 凭据时 Redfish 明确跳过，不阻断环境备份。
软件版本与编译构建链 software-versions.txt、build-system.txt，兼容文件 compiler-runtime.txt gcc/g++/clang/cmake/make/ninja/go/rust/python/java/docker/containerd/runc/podman/mysql/mysqld/nginx/redis 等版本；不存在则记录 missing-command。 逐项执行 --version、go version、java -version、nginx -v 等只读版本命令。 支撑编译优化、运行时差异、应用构建环境判断。
虚拟化线索 virtualization.txt 虚拟化类型、CPU hypervisor 标记、/proc/1/cgroup。 systemd-detect-virt、grep -i hypervisor /proc/cpuinfo、cat /proc/1/cgroup 判断 baremetal / VM / container，并影响 perf/PMU 可用性判断。
运行环境类型 environment-type.txt ENV_TYPE 和 RUNTIME_HINT。 脚本内检测：读取 /proc/1/cgroup，或调用 systemd-detect-virt。 标识当前采集发生在宿主机、虚拟机、容器或未知环境。
容器资源限制 container-limits.txt 容器可见 cpuset 与内存限制。 /sys/fs/cgroup/cpuset.cpus.effective、/sys/fs/cgroup/cpuset/cpuset.cpus、/sys/fs/cgroup/memory.max、/sys/fs/cgroup/memory/memory.limit_in_bytes 核对目标规格约束，避免把容器资源边界误判为整机能力。
内核配置 / 内核参数 / 启动参数 kernel-config.txt 内核启动参数、当前内核配置、模块列表。 cat /proc/cmdline、zcat /proc/config.gz、/boot/config-$(uname -r)、lsmod 判断内核 feature、启动参数和补丁/配置清单匹配情况。
采集过程审计 command-manifest.txt 每个命令的执行时间、命令文本、成功/失败/跳过状态。 脚本内部 log_manifest 记录。 用于报告可追溯性、失败命令解释和降级采集说明。
优化时间线 optimization-timeline.txt backup_start、backup_end、manifest 路径。 脚本内部时间戳。 用于本轮 run_id 证据新鲜度和后续耗时统计。
采集说明 README.txt 备份目录说明、预期产物清单、执行记录规则。 脚本生成静态说明。 方便人工查看环境备份目录结构。

当前环境诊断项与判定设计

环境诊断脚本会读取备份目录、当前系统文件、可选 reference 问题集和可选内核补丁清单，生成 environment_diagnosis JSON。当前核心诊断项如下。

诊断项 诊断内容 采集 / 检查方式 状态输出 失败或降级影响
历史 Reference 问题集回归 检查历史问题集中提到的参数或配置是否已经正常。 通过 --reference-issues 输入 JSON。支持 file_equals、file_contains、sysctl_equals、command_contains、backup_file_contains、backup_file_not_contains。 not_present、invalid、passed、failed、degraded 高严重级别失败会进入 blocked_items；缺失问题集时标记 not_present 并跳过，不报错。
BIOS 是否高性能配置 确认 BIOS/固件是否偏性能取向，关注 Power Profile、C-State、频率策略、NUMA/Node Interleaving、内存频率和 RAS 选项。 诊断脚本读取备份中的 bios-info.txt 和 CPU 信息搜索 performance / maximum performance / hpc 等关键词，以及 powersave / balanced / energy saving 等节能关键词。环境备份脚本已支持通过 Redfish 额外生成 bios-redfish-bios.json 和 bios-redfish-bios-settings.json。 passed、failed、degraded 只看到 BIOS 版本但看不到策略字段时标记 degraded；发现节能/均衡关键词时标记 failed，提示补充 BMC/Redfish 或 BIOS 截图。
Perf / PMU 采集能力 检查是否具备后续 perf、topdown、火焰图、硬件 PMU 事件采集条件。 检查 perf 是否存在；读取 /proc/sys/kernel/perf_event_paranoid、/proc/sys/kernel/kptr_restrict、/proc/self/status capabilities；检测 baremetal/VM/container；执行 perf list 和 perf stat -e cycles,instructions -- true smoke test。 passed、failed、degraded perf 缺失、权限不足、容器未映射 perf/PMU、VM 未暴露 PMU、硬件事件不可见都会阻断或降低后续 CPU 深挖置信度。
内核补丁是否齐全 检查目标场景要求的内核补丁或内核配置是否存在。 通过 --kernel-patch-manifest 输入 JSON。支持 uname_contains、os_release_contains、kernel_config_enabled、kernel_config_equals、file_exists、file_contains、command_contains、backup_file_contains。 not_applicable_or_unknown、invalid、passed、failed、degraded 无补丁清单时标记 not_applicable_or_unknown 并降级提示；高严重级别补丁缺失会进入 blocked_items。

Perf / PMU 细分检查项

字段 当前检查内容 来源
perf_command_status perf 命令是否存在。 shutil.which("perf")
perf_permission_status 当前用户是否 root，或是否具备 CAP_PERFMON / CAP_SYS_ADMIN。 os.geteuid()、/proc/self/status
runtime_environment 运行环境是 baremetal、VM、container 还是 unknown。 /.dockerenv、/proc/1/cgroup、systemd-detect-virt、备份文件。
perf_event_paranoid 内核 perf 权限限制级别。 /proc/sys/kernel/perf_event_paranoid
kptr_restrict 内核符号地址解析限制。 /proc/sys/kernel/kptr_restrict
pmu_event_status cycles、instructions、cache-misses、branches 等硬件事件是否可见。 perf list
perf_smoke_test_status 最小硬件事件采集是否可运行。 perf stat -e cycles,instructions -- true
capabilities 有效 capability 位图及 CAP_PERFMON、CAP_SYS_ADMIN、CAP_SYS_PTRACE 布尔值。 /proc/self/status 的 CapEff

环境诊断结果确认门控

诊断报告不是自动放行条件。Agent 需要把环境诊断摘要展示给用户确认，特别是 BIOS、perf/PMU、内核补丁、历史 reference 问题和降级证据。确认状态写入本轮 run manifest，并作为服务健康检查和基线性能确认的前置条件。

确认状态 含义 后续动作
pending 诊断已生成，但用户尚未确认。 停在环境诊断确认门控；只允许补采、解释诊断或生成 blocked/degraded 草稿。
confirmed 用户接受当前诊断结论和降级风险。 允许进入服务健康检查、目标实例身份确认和后续基线性能确认。
rebuild_required 用户要求补采或重建环境备份/诊断。 回到环境备份或诊断步骤，生成新的证据路径和诊断 JSON。
rejected / blocked 用户不接受诊断结论，或存在必须先修复的阻塞项。 输出当前阻塞报告；不得建立正式基线，也不得生成候选优化 skill 列表。
确认记录建议包含 confirmed_by、confirmed_at、confirmation_notes、accepted_degraded_items 和 remaining_blocked_items，用于最终报告和案例归档追溯。

诊断输出结构

当前诊断输出写入用户指定的 --output JSON 文件。主流程要求报告和案例归档保留该诊断结果，并在失败或降级时说明风险和最小补充证据。

输出字段 含义
schema_version、created_at 诊断结果 schema 版本和 UTC 生成时间。
status 环境诊断总状态：passed、failed 或 degraded。
reference_issue_set_status 历史 reference 问题集检查状态。
bios_performance_status BIOS 高性能配置检查状态。
perf_pmu_status perf/PMU 采集能力检查状态。
kernel_patch_status 内核补丁清单检查状态。
reference_issue_checks、kernel_patch_checks 逐项检查结果，包含 id、描述、严重级别、状态、实际值和错误信息。
bios_performance_findings、perf_pmu_findings BIOS 与 perf/PMU 的发现列表。
perf_pmu_checks perf 命令、权限、运行环境、paranoid、kptr、PMU 事件、smoke test、capabilities 的结构化检查结果。
findings 按 reference_issues、bios_performance、perf_pmu、kernel_patches 分类保留原始诊断结果。
blocked_items、degraded_items 阻塞项和降级项，用于决定是否停在环境诊断门控或继续但降低后续结论置信度。
evidence_paths 证据路径，当前至少包含环境备份目录。
next_steps 面向用户的补采集、权限修复或 BIOS 证据补充建议。
environment_diagnosis_confirmation_status 用户对诊断结果的确认状态：pending、confirmed、rebuild_required、rejected 或 blocked。
environment_diagnosis_confirmation_notes 用户确认意见、接受的降级项、要求补采的证据和阻塞项处置说明。

设计要求项 当前实现状态 建议补齐方向
GPU / NPU / 加速卡 未专门采集 主流程要求记录 GPU/NPU，但环境备份脚本当前没有独立 gpu-info.txt 或 NPU 工具采集。 按平台补充 nvidia-smi、rocm-smi、厂商 NPU 工具、PCIe 拓扑和驱动版本。

BIOS Redfish 当前配置 已支持 用户提供 BMC IP/用户/密码时采集 Redfish BIOS JSON；本次 target-920-55 未提供 BMC 信息，因此报告中标记 skipped。 开始采集前询问用户；自动化运行时可传 --bmc-host、--bmc-user、--bmc-pass 或设置环境变量。
网卡驱动、队列、中断、ring/coalesce 已增强 当前采集 ethtool -i/-k/-g/-c、RPS/XPS、IRQ affinity；逐接口 ethtool 带短超时保护。 后续可继续补充 ethtool -S 统计，避免环境备份文件过大时可做可选开关。
磁盘性能与 IO 调度 已增强 当前采集 lsblk -O、rotational、TRAN、型号、厂商、/sys/block/*/queue 调度器和队列参数。 后续可补充运行期 iostat -x，但这更适合瓶颈识别阶段而不是静态环境备份。
CPU governor / 频率 / C-State OS 侧证据 已增强 当前采集 cpupower frequency-info、scaling_governor、当前频率和 min/max 频率。 后续可补充 /sys/devices/system/cpu/cpu*/cpuidle C-State 细节。
可回退基线 manifest 部分具备 当前有备份目录和命令 manifest，但没有独立 restore_baseline_manifest 文件。 生成包含采集路径、关键配置快照、可还原项、不可还原项和人工恢复步骤的 manifest。
