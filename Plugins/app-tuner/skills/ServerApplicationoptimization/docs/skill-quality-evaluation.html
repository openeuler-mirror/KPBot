2. 好 Skill 的定义

Skill 的本质是给 Agent 使用的专业化操作手册。它应该把通用模型不知道、 容易忘、容易做错或需要稳定复用的流程沉淀下来，而不是把普通常识和长篇背景塞进上下文。S1

核心判断： 这个 skill 是否能让一个新的 Agent 在少量上下文下，更稳定、更安全、更快地完成同类任务。
重要边界： 自动化工具只能发现结构、格式、脚本和部分安全问题，不能替代真实任务验证和人工安全评审。

好的 skill 通常具备这些特征

来源：主要来自通用 skill 创建规范，安全项来自 skill vetting 规范，项目维护项结合本仓库实践整理。S1S3S4

  • 触发准确：name 简短清晰，description 明确说明能力、适用场景和典型触发语。
  • 上下文节约：SKILL.md 只保留核心流程，详细资料放入 references/ 并按需读取。
  • 流程可执行：步骤、输入、输出、停止条件、降级路径和用户确认点都写得清楚。
  • 自由度合适：开放型判断用原则和启发式，脆弱或高风险操作用脚本和强约束。
  • 验证闭环：提供样例任务、测试命令、预期产物和报告自检项。
  • 资源克制：只包含对任务有直接帮助的脚本、引用和资产，不放无关 README、历史说明或冗余材料。
  • 安全默认：最小权限、先 dry-run、先只读采集、危险动作显式确认，敏感数据不进入报告。
  • 可维护：主源、兼容入口、子 skill、脚本和文档之间不会漂移，变更后有回归检查。

常见差 skill 信号

来源：由 skill 创建规范中的反模式、安全 vetting 红线和本仓库性能优化实践归纳。S1S3S4

  • description 含糊，Agent 不知道什么时候该触发，或者过宽导致误触发。
  • SKILL.md 太长，把背景、范例、设计过程、用户文档全部混在一起。
  • 只给原则，没有输入输出契约、检查点、失败处理和可执行命令。
  • 脚本没有测试方式，依赖没有声明，失败时只报错不降级。
  • 要求读取密钥、浏览器会话、SSH 目录或系统文件，但没有充分理由。
  • 直接执行破坏性命令、远程命令或安装命令，没有权限边界和用户确认。
  • 没有样例任务，无法判断这个 skill 是否真的能提升 Agent 表现。

3. 100 分评分模型

建议把 skill 质量评审拆成八个维度。80 分以上通常可进入试用， 60 到 79 分需要带问题试用，低于 60 分不建议合入或安装。 八个维度来自来源 S1、S3、S4；具体权重和分段阈值是本文为团队评审沉淀的评分模型，不是上游工具的内置标准。S1S3S4

维度 权重 优秀标准 扣分信号
触发与定位 15 名称符合 kebab-case，描述包含能力、触发场景、文件类型或任务边界。 描述像口号；适用范围过宽；与其他 skill 重叠但没有边界。
渐进披露 15 SKILL.md 简洁，引用文件按任务路由，长材料不默认加载。 正文堆砌大段资料；引用文件没有读取时机；重复信息散落多处。
流程与契约 15 有清晰步骤、输入输出、门控、完成条件、失败处理和回退路径。 只有建议清单；Agent 需要自行猜测顺序和交付物。
可验证性 12 包含样例任务、验证命令、预期文件、报告字段和回归检查方式。 无法用一个新会话独立测试；没有成功标准。
脚本与工具质量 10 重复或脆弱动作脚本化，参数清晰，依赖少，错误信息可行动。 脚本不可执行；硬编码路径；安装依赖不说明；没有 dry-run。
安全性 18 最小权限、敏感数据保护、外部网络可解释、危险动作需确认。 读取密钥、访问 cookie、混淆代码、未知网络请求、sudo 或系统修改。
兼容与维护 8 主源明确，兼容入口保持同步，依赖和平台差异有说明。 多个入口内容不一致；文档指向不存在文件；命令只能在作者机器运行。
使用体验 7 Agent 能快速判断下一步，缺信息时知道问什么，输出对用户可读。 提示语啰嗦；过度询问；输出格式不稳定；报告缺少决策口径。
总分 结论 建议动作
90-100 高质量，可作为团队标准样例。 合入后保留回归用例，后续重点防止文档和脚本漂移。
80-89 可用，仍有局部优化空间。 允许试用，记录真实任务反馈并补齐薄弱项。
60-79 需要整改，不能直接视为稳定能力。 先修复触发、流程、验证或安全问题，再进入正式场景。
0-59 风险较高，不建议安装或合入。 重新定义任务边界，必要时拆分 skill 或删除高风险资源。

4. 推荐评审流程

来源：前半段来自 skill 创建流程和验证要求，安全审查步骤来自 skill-vetter 协议。S1S3

  1. 先看来源：确认作者、仓库、版本、更新时间、变更记录和是否来自可信渠道。
  2. 跑结构检查：验证 SKILL.md、frontmatter、命名、描述长度和必要目录。
  3. 读触发描述：判断 Agent 仅凭 name 和 description 是否知道何时使用。
  4. 读正文和引用：检查是否遵守渐进披露，是否有明确步骤、门控、输入输出和失败路径。
  5. 审脚本和依赖：逐个查看脚本是否安全、可参数化、可测试，依赖是否声明。
  6. 做安全扫描：搜索敏感文件访问、未知网络请求、混淆代码、危险命令和凭据泄露。
  7. 做 forward test：开一个新上下文，只给 skill 和真实任务，让 Agent 按 skill 完成，观察是否能泛化。
  8. 产出评审报告：给出分数、风险等级、阻塞问题、建议整改项和是否允许安装或合入。
Forward test 要点： 不要把预期答案、你的诊断或修复建议告诉测试 Agent。给它原始任务和 skill， 看它是否能独立找到正确路径，这比让它“评价 skill”更能暴露真实问题。

6. 安全审查清单

安全性应该作为独立门禁，而不是评分里的一个普通加分项。只要出现不可解释的高危行为， 即使其他维度分数很高，也应暂停安装、合入或执行。S3

必须回答的问题

问题 期望答案
skill 来自哪里，作者是否可信？ 来源、作者、版本、更新时间和用途都可追溯。
它需要读写哪些文件？ 只访问完成任务所需的工作区文件或明确资源。
它会运行哪些命令？ 命令可解释、可复现，危险命令需要用户确认。
它需要网络访问吗？ 访问目标、协议和数据内容明确，没有未知外传。
它是否处理凭据、客户数据或生产环境？ 最小权限、脱敏、只读优先，有审计和回退方案。

出现以下红线时应立即拒绝

  • 向未知 URL 或 IP 发送文件、环境变量、对话内容、密钥或日志。
  • 读取 ~/.ssh、~/.aws、浏览器 cookie、token、密码文件，且没有明确任务必要性。
  • 使用 eval、exec、base64 解码或混淆脚本来执行外部输入。
  • 请求 sudo、修改系统目录、安装未知包或改变安全配置。
  • 默认执行删除、格式化、重启、远程写入、生产变更等破坏性动作。
  • 把客户路径、IP、账号、业务数据或压测原始日志直接写入对外报告。

风险分级

等级 特征 处理建议
低风险 纯文档、格式转换、只读分析，无外部网络和敏感数据。 基础审查通过后可试用。
中风险 读写工作区文件、调用本地脚本、访问公开 API 或处理内部资料。 完整代码审查，限制权限，保留验证记录。
高风险 涉及凭据、SSH、数据库、生产系统、远程执行或系统参数。 必须人工批准，默认 dry-run，变更前写明回退方案。
极高风险 要求 root、安全配置变更、密钥读取、未知网络外传或混淆执行。 默认拒绝，除非完成专门安全评审和隔离验证。
安全门禁建议： 对服务器性能优化类 skill，任何真实环境变更都应区分只读采集、dry-run、低风险在线变更、 需重启变更和需重新编译变更。不能把候选动作直接当成可执行命令。
