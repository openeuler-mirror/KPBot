Skill 门控执行 Workflow

本图按当前 skills/kpbot-app-tuner 的门控逻辑绘制， 用于 review Agent 是否严格自顶向下执行，并在异常、历史产物混入或证据失效时优先输出当前阻塞状态。

1. 执行流一图总览

主流程分为 Phase 1 环境基座、Phase 2 瓶颈识别与候选优化 skill 列表、Phase 3 迭代验证和收尾。 红色分支代表必须停止并输出阻塞/降级报告的路径，不能继续引用历史产物伪装成当前结论。

Phase 1输入、run_id、备份、诊断、健康、基线
Phase 2瓶颈识别、证据采集、候选 skill 列表
Phase 3执行验证、报告、review、还原、归档
异常分支停止下游，生成 blocked/degraded 报告
0. 启动门控

读取架构、输入契约、检查清单；初始化 overall_progress。

1. 用户界面层确认

确认测试场景、目标规格、压测方法、Agent 操作模式、变更范围、耗时和收益统计口径。

2. 本轮运行身份

场景和操作边界确认后生成 current_run_id、current_run_started_at、current_run_manifest；没有 run_id 的产物只算历史记录。

3. 环境备份

生成 environment_backup_dir、dependency_status、restore_baseline_manifest。

4. 环境诊断

检查历史 reference 问题集、BIOS 高性能配置、perf/PMU 采集能力、内核补丁齐全性；阻塞项停在 environment_diagnosis_completed。

环境诊断是否存在阻塞项？

例如关键依赖不可用、内核补丁缺失且影响测试可信度。

是：停止在环境诊断门控
  • overall_progress.status=blocked
  • blocked_gate=environment_diagnosis_completed
  • 输出修复建议，不进入服务健康和基线。
否：继续
  • 证据不足但非阻塞时标记 degraded。
  • 进入诊断结果用户确认。
5. 环境诊断结果确认

向用户展示 blocked_items、degraded_items、BIOS/PMU/内核补丁风险和补采建议；确认状态写入 environment_diagnosis_confirmation_status。

用户是否确认环境诊断结果？

必须满足 environment_diagnosis_confirmation_status=confirmed，才允许进入服务健康和基线前置流程。

否：停止在诊断确认门控
  • environment_diagnosis_confirmation_status=pending|rejected|rebuild_required|blocked
  • 不得进入服务健康、目标实例身份确认和基线性能确认。
  • 按用户意见补采、重建诊断或输出当前阻塞报告。
是：进入服务健康
  • 记录确认人、确认时间和接受的降级项。
  • 本轮 run manifest 绑定诊断确认结果。
6. 服务健康检查 + 目标实例身份

端口/入口、协议 smoke test、认证权限、最小负载、PID/容器/二进制/配置/资源约束。

服务健康和目标实例是否确认？

必须满足 environment_diagnosis_confirmation_status=confirmed、service_health_status=passed 且 target_instance_identity.status=confirmed。

否：输出当前异常
  • 不得建立正式基线。
  • 不得进入瓶颈识别和候选 skill 列表生成。
  • 不得用历史报告替代当前失败。
是：进入基线
  • 记录健康检查证据。
  • 保留目标实例身份用于后续归因。
7. 目标规格下建立基线

先约束资源，再运行压测；记录原始日志、资源利用率、测试组网信心、测试用例信心。

用户是否确认基线？

未确认前不得进入候选优化 skill 列表生成。

否：回退重建
  • 回到场景输入、规格约束或基线测试。
  • baseline_confirmation_status=rebuild_required 或 blocked。
是：写入检查点
  • 写入 checkpoint_5.json。
  • baseline_confirmation_status=confirmed。
8. 历史记录使用门控

发现历史日志、报告、旧轮次时，仅记录为 discovered_unconfirmed；用户确认前不能参与当前结论。

9. 性能采集与瓶颈识别

识别磁盘、网卡、内存、CPU、GPU/NPU、硬件规格瓶颈；低置信度先补采或降级说明。

是否存在可识别瓶颈？

unknown_bottleneck 或 no_active_bottleneck 不进入盲目调优。

否：进入报告
  • 输出无法识别或无活跃瓶颈结论。
  • 给出补采或下一步计划。
是：继续采集深度证据
  • 瓶颈分类进入候选列表准备。
  • 仍需通过证据新鲜度校验。
10. 基线后深度证据采集

按目标进程采集火焰图、perf data、热点函数、热点 DSO/so 排名、topdown L1 icache miss、线程切换、L3/LLC cache miss 等。

证据是否属于本轮且状态为 current？

校验 run_id、采集时间、目标 PID/容器/端口、是否混入历史产物。

否：证据门控阻塞
  • current_evidence_status=missing|stale|mixed|invalid
  • bottleneck_classification=blocked
  • 清空候选 skill 列表、候选池、收益和优化动作。
是：允许生成候选列表
  • current_evidence_status=current
  • 所有下游结论绑定同一 current_run_id。
11. 生成候选优化 skill 列表

读取 performance_signal_summary.json，根据热点 so、网络热点、L1 icache miss、线程切换和 L3/LLC miss 生成 candidate_skill_list。

  • 高热点第三方库 → performance-library-selection
  • 网络相关热点高 → network-optimization
  • L1 icache miss 高 → compiler-optimization，重点 PGO/LTO
  • 线程切换高且 L3/LLC miss 高 → cpu-affinity-optimization
12. 候选/coverage skill 分析

先执行 phase=evidence_candidate 的 skill；候选完成后追加未命中的主优化 skill 作为 phase=coverage，最终所有主优化 skill 都有结论。

13. 串行执行验证

每轮一个执行验证 subagent 负责当前 skill 的实施、复测、回退和结果记录；危险动作必须为 approved_execute。

当前 skill 是否应停止？

单个 skill 最多尝试 5 轮；若 5 轮收益均小于 1%，或候选动作耗尽/风险超界/用户停止，则停止当前 skill。

停止当前 skill
  • 记录 skill_stop_reason。
  • 继续执行 candidate_skill_list 中下一个候选或 coverage skill。
继续当前 skill
  • 接受有效动作或回退负收益动作。
  • 进入下一轮验证。
候选列表循环：先完成证据命中的候选 skill，再完成 coverage skill；所有主优化 skill 完成、停止或阻塞并说明原因后进入报告。
14. 报告输出

正常路径输出最终优化报告；阻塞路径输出当前异常、失败证据、补采清单和下一步，不输出正式收益。

15. Review、环境还原、案例归档

确认最终有效配置、已回退项、残留风险；按 manifest 还原环境；归档 case_archive.json。

2. 输入输出目录与文件结构

当前 skill 的脚本允许调用方传入任意输出目录；为保证 run_id、证据新鲜度和案例归档一致，推荐统一落盘到 output/kpbot-app-tuner/<current_run_id>/。所有正式结论只引用该目录下、且 current_run_id 一致的产物；历史日志和旧报告必须进入 history/ 或待确认区，不能直接覆盖当前结论。

report/、restore/、archive/ 主 Agent 收尾阶段 用户、后续案例复用 正常路径输出最终报告；阻塞路径输出 blocked/degraded 报告。当前 generate_report.py 生成 Markdown，HTML 报告可由主 Agent 或后续转换脚本生成；归档 case_archive.json 作为后续案例输入。

3. 数据流与控制流

数据流描述文件如何被生产和消费；控制流描述哪些状态允许继续。两者必须同时满足：即使文件存在，只要门控状态未确认或 current_run_id 不一致，也不能进入下游。

输入归一化 → run manifest

用户场景、操作授权、BMC/Redfish 选择和目标规格先写入 inputs/，再生成本轮 current-run-manifest.json。

环境备份 → 环境诊断 → 用户确认

env/backup/ 是诊断 JSON 的输入；诊断摘要必须经用户确认后，才能进入服务健康和基线。

基线 → 证据 → 候选列表

已确认基线和当前证据生成 performance_signal_summary.json，再驱动候选 skill task manifest。

候选任务 → 子 skill 结果 → 候选池

各 subskill 只读任务包并输出 JSON 结果；合并脚本校验 run_id、证据状态和候选覆盖后生成 candidate_pool.json。

迭代验证 → 收益统计

每轮动作都绑定前后验证、回退计划和收益摘要；负收益或风险超界动作不得沉淀为有效配置。

报告 → 还原 → 归档

最终报告引用当前 run 目录内证据；restore 执行后写结果，case archive 只归档已确认或明确标注降级的事实。

4. 门控状态字段

门控 关键字段 通过条件 失败时动作
启动门控 overall_progress 已读取架构、输入契约、检查清单，并持续展示进度。 不得进入调优；补齐入口上下文。
用户界面层确认 scenario_input_state、agent_action_confirmation 场景、目标规格、压测方法、操作模式和变更范围已确认。 不得生成正式 run manifest；继续补齐输入和权限边界。
本轮运行身份 current_run_id、current_run_manifest 本轮所有产物可追溯到同一 run_id。 产物标记为历史记录；报告转为 blocked/degraded。
环境备份与诊断 environment_backup_dir、environment_diagnosis、perf_pmu_status 备份可还原，历史问题集、BIOS、perf/PMU、内核补丁诊断阻塞项为空或已明确降级。 停止在环境诊断门控，输出修复项。
环境诊断结果确认 environment_diagnosis_confirmation_status、environment_diagnosis_confirmation_notes 用户确认诊断结果，且明确接受降级项或补采处置。 不得进入服务健康、目标实例身份确认和基线性能确认；按用户意见补采、重建诊断或输出阻塞报告。
服务健康与实例身份 environment_diagnosis_confirmation_status、service_health_status、target_instance_identity.status 环境诊断已确认，服务可压测，目标实例唯一且可归因。 不得建立基线；不得引用历史报告替代当前失败。
基线确认 baseline_confirmation_status 用户确认目标规格、测试口径和基线指标。 回退到场景输入、规格约束或基线测试。
历史记录确认 historical_records_status 用户明确确认历史材料适用于本轮，且限定使用范围。 只能放入待确认材料章节，不进入收益和动作结论。
瓶颈识别 bottleneck_classification、bottleneck_confidence 基线已确认，瓶颈证据充分且分类可信。 补采、降级说明或进入无可识别瓶颈报告。
证据新鲜度 current_evidence_status 证据状态为 current，run_id、时间、目标身份一致。 清空下游正式结论，输出补采清单。
候选 skill 列表 candidate_skill_list、performance_signal_summary、candidate_pool 根据采集信息生成候选和 coverage skill，候选动作包含验证和回退。 缺证据则 blocked；不生成猜测性动作。
迭代执行 per_skill_iteration_state、iteration_decision 串行验证，收益和回退清楚，危险动作已确认。 停止当前 skill、回退动作或全局停止。

5. 异常优先级

当前异常优先于历史成功

  • 环境诊断未确认时，不允许进入服务健康或基线性能确认。
  • 当前服务健康失败时，不允许引用历史报告继续调优。
  • 当前证据失效时，不允许保留旧候选池或旧收益。
  • 当前 run_id 缺失时，所有下游结论只能是 blocked/degraded。

历史产物默认未确认

  • historical_records_status=discovered_unconfirmed
  • 用户确认前只能做只读摘要。
  • 不得进入收益表、候选动作、停止规则和最终优化结论。

正常路径输出

  • 同一 current_run_id 贯穿基线、证据、候选列表、轮次和报告。
  • current_evidence_status=current
  • 每轮有验证结果、风险、回退和收益归因。

阻塞报告输出

  • 展示 blocked_gate、失败证据和修复建议。
  • bottleneck_classification=blocked 或 not_entered。
  • 候选 skill 列表、候选池、正式收益和优化动作为空。

6. Review 检查点

Review 问题 符合预期时应看到
是否可能跳过上游门控直接进入优化？ 不应可能。环境诊断结果确认、服务健康、目标实例、基线确认、证据新鲜度均是硬门控。
异常时是否会引用历史产物？ 不会作为正式结论。历史产物只能进入待确认章节，当前异常优先。
候选列表是否只执行命中 skill？ 不是。先执行证据命中的 evidence_candidate，再执行未命中的 coverage skill，最终所有主优化 skill 都有结论。
执行验证是否可能并发改环境？ 不应可能。每轮只允许一个执行验证 subagent 修改环境。
单 skill 低收益时如何处理？ 单个 skill 最多 5 轮；若 5 轮收益均小于 1% 则停止该 skill，并继续候选列表中的下一 skill。
最终报告是否区分正常与阻塞？ 应区分。阻塞报告不输出正式收益、候选动作或优化结论。
